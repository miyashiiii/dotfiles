# 個人設定（全プロジェクト共通）

## 破壊的操作の安全性

- **`git push` は明示的な許可があるまで行わない**。コミットまでは OK だが、push は必ずユーザーの指示を待つ。PR 作成時も同様。一度許可されても、その許可は該当コミットのみに有効。次のコミット以降は改めて許可を待つ。
- **PRやブランチを気軽に作り直さない**。force push・ブランチ切り直し・PR 閉じて再作成などの破壊的操作は、事前に選択肢と影響を提示してユーザーの判断を仰ぐ。自分から「作り直しましょう」と提案しない。
- **削除・上書き・リセットなど取り消し困難な操作は、Git に限らず必ず事前確認する**。

## コードスタイル

- **略称を使わない**。識別子・コメント・会話すべてにおいてフルネームで書く。

## ドキュメント・Markdown の書き方

- **接頭辞付きの箇条書き（`- **項目**: 説明…` の形）を多用しない**。読みにくくなる。項目ごとに整理したくなったら、(a) 見出し（`###`）を立てるか、(b) 箇条書きをネストさせる。
- **ドキュメント内で略語・記号コードを使わない**（例: Q1 / P3 / FS-2 などの内部コード）。読み手が一覧を引かずに理解できるよう、普通の日本語で書く（「再アクティベートなし」「落とし物連携」等）。
- 一般的でない言い回しを避ける（例: スケジュールの意味で「暦」と書かない）。

## CLAUDE.md の育て方

- ユーザーが「こういうルールで動いて」「こういう感じでコミュニケーションして」と依頼した場合、対応したうえで「CLAUDE.md に反映しますか？」と確認する。

## 言語・コミュニケーション

- コメント・コミットメッセージ・PR・テスト名は**日本語**で記載する。
- 絵文字は使わない（ユーザーが明示的に求めた場合を除く）。
- 回答は簡潔に。不要な前置き・まとめは省く。

## Effort / モデル / clear のうながし

タイミングが来たら軽く一言うながす（押し付けない）。

- **effort / モデル**: 重い思考（設計・複数ファイル横断・整合性/敵対監査・難デバッグ）に入る前は上げを、軽作業（文面修正・定型コミット・機械置換）が続くときは下げを、適宜提案する。**提案する前に基準 `~/.claude/refs/model-effort-policy.md` を読み**、どのレベル/モデルにするかはその基準に沿って決め、必ず「いまの値」と「おすすめの変更先」を添える。
- **clear**: タスクが一区切りで引き継ぎ不要なとき・別テーマに切り替わるとき・文脈が肥大化したときに促す。

## 機密情報

- 鍵・トークン・パスワード・個人情報をコード/コミット/出力/ログに残さない。`.env` 等の中身を会話やコミットに貼らない。必要時はプレースホルダや参照で示す。

## 進め方（計画・実装・完了）

- **計画先行**: 大規模・不可逆な変更（設計判断・多数ファイル横断・取り消しにくい操作）はプランモードで計画を提示し、承認を得てから実装する。それ以外は通常モードで進める（必要なら着手前に軽い見通しを述べる）。
- **完了報告**: コード変更後は、そのプロジェクトの lint / test を通し、**さらに実際に動かして動作確認してから**「完了」と報告する。落ちた場合は出力とともに正直に伝え、未検証で「完了」と言わない。
  - UI が絡む動作確認は難しいことがある。**一度試したうえで困難ならスキップ可**。セッションの文脈上すでに UI 動作確認を諦めている状況でもスキップ可。
